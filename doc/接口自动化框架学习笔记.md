### basetoken并发修改

以前 BaseTest在接口执行前登录一次，将token设置为全局静态变量，示例代码：

```java
public static String token = "";//测试类的全局token
```

```
// BaseTest.java (v1)
public static String token = "";  // ⚠️ 静态变量 = 所有线程共享同一块内存

@BeforeAll
void setAll() {
    // ⚠️ 修改全局静态配置 = 所有测试类同时被影响
    RestAssured.baseURI = "http://localhost";
    RestAssured.port = 8080;
    
    // 登录获取 token，赋值给静态变量
    new Auth().login(...);
    BaseTest.token = BaseInterface.getToken();  // ⚠️ 线程A/B/C 都在写同一个变量
}
```

所有测试类共享同一份状态 → 并发时互相覆盖token → 用例失败。

比如说， 线程1 (AuthLoginTest) 调用login获取tokenA，而 线程2 (UserPageTest)调用 login() 获取 token_B，线程1执行用例会拿到tokenB，但实际上TokenB可能是另一个账号的权限，权限不足导致用例1失效了，用例会报401权限错误。

**你的直觉里 "用 tokenB 也能跑" 这个想法**——前提是 tokenB 一直存在。但**测异常用例时,token 可能变空、变成无效值**。

而使用 RequestSpecFactory 可以并发执行，每次请求新建独立实例 → 状态隔离 → 安全并发

> "v1 把 token 设成 static,这个字段在 JVM 方法区只有一份,所有测试类共享。单线程跑没问题,但开 JUnit 并行后,多个测试类实例同时读写这个字段会产生**竞态**——一个测试类登录失败把 token 设空,另一个测试类拿空 token 调接口就 401 了。改成实例字段后,每个 Test 类实例在堆上有自己的副本,互不影响。

### **并发处理不好的影响：**

- 静态 token 被互相覆盖。`static` 意味着**整个 JVM 只有一份**。所有测试线程读写的是同一个内存地址。

- RestAssured 全局静态配置 (v1定义api时的代码，雷点，要是并发跑时，可能会请求到错地方)

  ```java
  public Response login(...) {
      RestAssured.basePath = "/sys/auth/login";  // ← 这才是大坑
      ...
  }
  
  public Response logout(...) {
      RestAssured.basePath = "/sys/auth/logout"; // ← 改了同一个全局字段
      ...
  }
  ```

- 数据库清理互相伤害 (一个进程清理了数据库，另一个进程需要验证数据库信息但是没信息了)

  ```
  jdbcTemplate.update("delete from sys_user where username != 'admin'");
  ```

工程界共识:**能不共享就不共享**。"线程安全的代码"不如"没有共享状态的代码"。

RequestSpecFactory 解决了并发竞态问题。因为每个线程有自己的:

- ✅ 自己的 RequestSpecification 实例
- ✅ 自己的 token(通过参数传进来,不读全局)
- ✅ 自己的 basePath(写在自己的 spec 里)

数据库清理 加入测试数据标识，只删除自己测试类的数据，这样就不会误删其他测试类的数据了。

**两个工程原则**:

1. **数据"自带身份"**:看 username 就知道是谁的,不用查代码
2. **操作"最小作用域"**:只操作自己产生的数据,不越权碰别人

### 细节 1:`private final` 的妙用

```java
private final RequestSpecification baseSpec;
```

- `private`:外部拿不到,**保证模板不被外部弄脏**
- `final`:**字段只能赋值一次**(在构造函数里),之后整个生命周期不会被换掉

### 细节 2:为什么 baseSpec 不含 token?

你可能想:为啥不一开始就把 token 也烤进 baseSpec?

因为:

- baseSpec 是**整个 JVM 共用的**模板,一份就够
- 但**token 是每个测试类自己的**,不能进模板

所以 token 走参数传:

```java
public RequestSpecification auth(String token) {
    return given().spec(baseSpec).header("Authorization", token);
    //                            ↑ token 在这里现场加,不污染 baseSpec
}
```

这是分层设计:**不变的进模板,会变的走参数**。

### 细节 3:`PrintStream.nullOutputStream()` 这招

```java
PrintStream silent = new PrintStream(PrintStream.nullOutputStream());
...
.addFilter(new RequestLoggingFilter(LogDetail.ALL, silent))
```

意图:**保留 Filter,但不让它打到控制台**。

为什么这么干?

- RestAssured 的 LoggingFilter 默认全打到 `System.out`,本地开发能看,**CI 里会刷屏几千行**,真有问题反而找不到
- AllureRestAssured 这个 Filter 会**自动把请求/响应附加到 Allure 报告**——失败时点开报告就能看完整信息
- 所以日志 Filter 留着是为了**保留这个"管道"**,但输出到 `/dev/null`,实际信息靠 Allure 报告看

简单理解:**关掉嘴(控制台),开着耳朵(报告)**。

注释里也告诉你了:想恢复控制台日志就把 `silent` 改成 `System.out`。

### 细节 4:为什么 `unauth()` 也要走 `given().spec(baseSpec)`?

```java
public RequestSpecification unauth() {
    return given().spec(baseSpec);
}
```

不能写 `return baseSpec;`?

**不能**。**共享风险**:如果直接返回 baseSpec,调用方 `.header(...)` 一下就**改了模板本身**,后面所有人都受影响

所以 `given().spec(baseSpec)` 干的事是:"<u>基于模板新开一个,你随便改</u>"。

### JVM相关总结知识

**JVM 就是一个能运行 .class 文件的程序**

```
┌─────────────────────────────────────────┐
│  方法区 / Metaspace                      │  ← 类信息、static 字段在这
├─────────────────────────────────────────┤
│  堆 (Heap)                              │  ← new 出来的对象在这
├─────────────────────────────────────────┤
│  栈 (Stack) - 每个线程一个              │  ← 方法局部变量在这
└─────────────────────────────────────────┘
```

**JVM 是一个进程**,跑完就死,内存清空

**栈** = 每线程独享,放局部变量,**天然安全**

**堆** = 所有线程共享,放 `new` 的对象

**方法区** = 放类信息和 `static` 字段,**全 JVM 一份**

**static 非 final 字段 + 多线程 = 大概率有 bug**

**final 字段(无论 static 与否)= 安全**(只要里面装的是不可变对象)

解决并发的最佳方法:**让线程不共享数据**(每次 new 新的)

解决办法有三种思路：

- 不共享⭐⭐⭐⭐⭐ 最简单最稳
- 不可变⭐⭐⭐⭐ 共享但只读,也安全
- 加锁同步⭐⭐ 能用但容易写错、性能差



## BaseTest

### 为什么 BaseTest 是 `abstract` 的?

为什么这么设计?

- BaseTest **本身没有具体测试方法**(没有 `@Test`)
- 它的存在意义是**给子类提供共享能力**(token、specFactory、jdbcTemplate)
- 直接 new 它没意义,JUnit 也不会跑

加 `abstract` 是在向调用者明示:"我是个父类,你别直接用我,你应该继承我"。这是面向对象的常见做法,叫**模板方法模式**。

### 字段为什么用 `protected`?

Java 四种访问修饰符:

| 修饰符          | 谁能访问    |
| --------------- | ----------- |
| `public`        | 所有人      |
| `protected`     | 同包 + 子类 |
| `(默认/包私有)` | 同包        |
| `private`       | 只有本类    |

### 为什么测试类使用 `@TestInstance(PER_CLASS)`

PER_CLASS 模式的核心好处:**让你能用实例字段代替静态字段**,从而摆脱"全局共享"。

### 对比v1 模板参数的处理

v1版本写法：

```java
public abstract class BaseTest {
    protected DocumentContext defaultData;   // ← 你以前的写法
}
class UserPageTest extends BaseTest {

    void beforeEach() {
```

#### v1问题1

哪怕是**单线程**,这个写法都会出 bug。**DocumentContext 是可变对象(mutable)**——`.set()` / `.delete()` **改的是它本身**,不是返回新对象。但是做的好的地方是，beforeeach每次会重置。

#### 并发污染

junit5有两种并发粒度：

```
类级别并发(不同类并行,同类内串行)

junit.jupiter.execution.parallel.mode.classes.default = concurrent
junit.jupiter.execution.parallel.mode.default = same_thread

方法级别也并发(同类内方法也并行)

junit.jupiter.execution.parallel.mode.default = concurrent
```

**类级别并发**下,`defaultData` 是实例字段,每个测试类有自己的实例,**反而没问题**(因为同一个类的方法还是串行的)。

**方法级别并发**下,问题就来了。多个测试方法开的线程在改同一个 DocumentContext 实例,数据会乱成一锅粥。

## ENV

该文件解决了一个核心痛点：**如何让同一套测试代码，在本地开发、CI 流水线、生产预演等不同环境下，自动读取不同的配置，且不需要修改代码。**

### 饿汉式单例模式设计

```JAVA
private static final Env INSTANCE = load();

public static Env get() {
    return INSTANCE;
}
```

**什么是单例?** 一个类全 JVM 只有一个实例。这里 Env 就是单例——配置文件读一次就够,不需要每次 new。

**为什么叫"饿汉式"?**

| 类型       | 何时初始化                              | 优点           | 缺点                    |
| ---------- | --------------------------------------- | -------------- | ----------------------- |
| **饿汉式** | 类加载时(JVM 启动加载这个类就立刻 init) | 简单、线程安全 | 哪怕没人用也会创建      |
| **懒汉式** | 第一次调用 get() 时                     | 按需创建       | 多线程下要加锁,容易写错 |

**饿汉式直接利用 JVM 的"类加载机制"**——JVM 保证一个类的 static 字段初始化是**线程安全的、且只会执行一次**(由 JVM 内部的 `ClassLoader.loadClass` 加锁实现),所以根本不用自己加锁。

懒汉式基础版本：

```java
private static Env instance;
public static Env get() {
    if (instance == null) {       // ← 多线程同时进入这行
        instance = load();         // ← 可能 new 出多个
    }
    return instance;
}
```

要写对懒汉式得用"双重检查锁定"+ `volatile`,代码变成这样:

```java
private static volatile Env instance;
public static Env get() {
    if (instance == null) {
        synchronized (Env.class) {
            if (instance == null) {
                instance = load();
            }
        }
    }
    return instance;
}
```

复杂、易错、性能差。

> **行业经验**:除非你的单例创建特别贵(比如要 30 秒),否则**永远用饿汉式**。这是 Joshua Bloch《Effective Java》里明确推荐的。



### 为什么优先级是：`-D` > `环境变量` > `配置文件` > `默认值`？

**这是工业级配置加载的标准做法**。覆盖优先级一般是:**临时 > 持久 > 默认**。

| 优先级                 | 来源          | 适用场景                  |
| ---------------------- | ------------- | ------------------------- |
| ① 系统属性 `-Dkey=val` | 启动命令行    | 临时调试,优先级最高       |
| ② 环境变量 `KEY=val`   | shell/CI 配置 | CI 流水线、容器、敏感信息 |
| ③ properties 文件      | 项目里的配置  | 团队共享的环境配置        |
| ④ 默认值               | 代码里硬编码  | 兜底,IDE 直接跑能不挂     |

这个顺序不是随机定的，而是基于 **“控制权的精确度”** 和 **“安全性”** 设计的工业标准。

我们可以用**“谁离用户最近，谁说了算”**来理解：

#### 第一级：系统属性 (`-D`) —— 【临时指令】

- **特点**：显式、即时、针对单次运行。
- 为什么最高？
  - 当你手动敲下 `-D` 时，代表你**此时此刻**明确希望程序这么跑。
  - 比如：平时连测试库，今天我想临时连一下生产库查个数据。我不想去改配置文件（怕忘改回来），也不想改环境变量（太麻烦）。我直接在启动命令加个 `-D`，用完即走。
  - **原则**：用户的**即时意图**高于一切静态配置。

#### 第二级：环境变量 (`ENV`) —— 【部署环境】

- **特点**：与机器/容器绑定，对代码不可见，安全。
- 为什么排第二？
  - 在 Docker、K8s、Jenkins 等现代部署环境中，密码、密钥等敏感信息**严禁**写在代码或配置文件里（因为会泄露到 Git）。
  - 环境变量是由运维人员或 CI 平台注入的，代表了**当前运行环境**的真实身份。
  - 它比配置文件更“真实”，因为配置文件可能只是开发时的模板，而环境变量才是生产环境的真相。

#### 第三级：配置文件 (`properties`) —— 【项目默认】

- **特点**：随代码提交，团队共享，方便开发。
- 为什么排第三？
  - 这是给开发者用的“默认值”。
  - 它方便大家拉下代码就能跑，不需要每个人去配环境变量。
  - 但它不够灵活（改完要重新编译或重启），也不够安全（不能存明文密码）。所以当有更高级的指令（环境变量或 `-D`）时，它应该让位。

#### 第四级：硬编码默认值 —— 【保底兜底】

- **特点**：写死在代码里。
- 为什么最低？
  - 这是最后一道防线。如果连配置文件都丢了，或者新同事刚拉代码还没配环境，程序至少能启动起来（虽然可能连不上数据库，但不会报 `NullPointerException` 崩溃）。
  - 它没有任何灵活性，所以优先级最低。

### try-with-resources 自动关流

```java
try (InputStream is = Env.class.getClassLoader().getResourceAsStream(classpathPath)) {
    if (is != null) {
        props.load(new InputStreamReader(is, StandardCharsets.UTF_8));
    }
} catch (IOException ignored) {
}
```

 `try (...)` 这个语法，会自动关闭括号里的资源，等价于在finally 里写了关闭流逻辑。

> 小习惯:**只要见到 InputStream / OutputStream / Connection / Statement 这些"资源",一律用 try-with-resources**。这是 Java 7 之后的强制最佳实践。

## JsonTemplateUtil 

```
把请求体存为 JSON 文件（template/user/save.json），测试时读取后按需改几个字段
优点：正常用例直接用模板；异常用例用 .set()/.delete() 改几个字段，代码非常简洁。
```

### 共享模板改进

#### ❌ v1 老写法

```java
public abstract class BaseTest {
    protected DocumentContext defaultData;     // ← 父类共享字段
}

class UserPageTest extends BaseTest {
    @BeforeEach
    void beforeEach() {
        defaultData = JsonPathUtils.jsonPathParseByFile("/template/user/page.json");
    }
    
    @Test void test01() {
        defaultData.set("$.page", 2);
        userApi.page(token, defaultData.json());
    }
}
```

问题:

- 字段在父类里,所有子类共享(看着像共享,实际每个实例独立,但**这种"看起来共享"的设计就是坏设计**)
- 跨方法可变,需要 `@BeforeEach` 兜底
- 不能并发

#### ✅ v3 推荐写法:每次方法内部加载,不当字段

```java
class UserPageTest extends BaseTest {

    private static final String TEMPLATE = "template/user/page.json";

    @Test
    void should_return_success_when_valid_params() {
        Map<String, ?> body = JsonTemplateUtil.load(TEMPLATE)
                .set("$.page", 2)
                .toMap();                // ← 链式 + 立刻消费

        userApi.page(token, body)
                .then().statusCode(200)
                .body("code", equalTo(0));
    }

    @Test
    void should_return_500_when_missing_page() {
        Map<String, ?> body = JsonTemplateUtil.load(TEMPLATE)
                .delete("$.page")
                .toMap();

        userApi.page(token, body)
                .then().body("code", equalTo(500));
    }
}
```

关键改进:

1. **DocumentContext 不再做字段**,只是方法内的局部变量
2. **每个测试方法独立加载模板**,天然隔离
3. **链式立即消费,不在方法间持续存活**
4. **局部变量在线程栈上,天然线程安全**

判断口诀:**变化频率越高,作用域应该越小**。

```
变化频率:不变 < 每类一次 < 每方法一次 < 每次调用一次
作用域: static 字段 < 实例字段 < 局部变量 < 方法参数
              ^                                          ^
              对应关系:左边匹配右边
```

你 v1 的 `defaultData` 是**"每方法变化"的高频量**,放到了**"实例字段"这个低频作用域**,错配了。这是它别扭的根本原因。

你 v1 这段代码看似简单,但展开能讲三层考点,每层都加分:

第一层:测试隔离(必懂)

> "v1 把 DocumentContext 设为父类字段,每个测试类的多个测试方法共享这个字段。如果某个方法 `.set()` 后忘了重置,后续方法拿到的就是脏数据,导致**测试结果与执行顺序耦合**。我用 `@BeforeEach` 强制每次重新加载模板才避免了这个问题,但这是治标不治本。v3 改成方法内局部变量,每次链式立即消费,根除了这层耦合。"

第二层:并发安全(进阶)

> "DocumentContext 这个类本身不是线程安全的,内部维护可变状态。如果开 JUnit 方法级并行,多个测试方法同时改这一个字段会有竞态。v3 把它从字段下沉到局部变量,放到线程栈上,天然隔离。"

第三层:不可变性思想(高阶)

> "可变共享状态是测试代码最大的反模式。v3 整体设计里,token、specFactory、jdbcTemplate 是 `@BeforeAll` 初始化后不再修改,DocumentContext 这种本身就可变的类则严格限定在方法作用域。这样无论 JUnit 跑串行还是并行,行为都是一致的。"

## BaseApi

### **模板方法模式**

是行为型设计模式的一种。它在一个**抽象类**中定义了一个算法的**骨架（流程）**，而将算法中的某些**具体步骤**延迟到**子类**中去实现。

教科书的写法。

```java
// 抽象父类：定义骨架
abstract class BeverageMaker {

    // 1. 模板方法 (最终方法，子类不能改流程)
    public final void makeBeverage() {
        boilWater();       // 步骤1：烧水 (所有饮料都一样，父类实现)
        brew();            // 步骤2：冲泡 (不同饮料不一样，子类实现)
        pourInCup();       // 步骤3：倒杯里 (所有饮料都一样，父类实现)
        addCondiments();   // 步骤4：加调料 (不同饮料不一样，子类实现)
    }

    // 具体方法
    private void boilWater() {
        System.out.println("烧开水...");
    }
    
    private void pourInCup() {
        System.out.println("倒入杯中...");
    }

    // 抽象方法 (钩子方法，强制子类实现)
    protected abstract void brew();
    protected abstract void addCondiments();
}

// 子类 A：咖啡
class Coffee extends BeverageMaker {
    @Override
    protected void brew() {
        System.out.println("用沸水冲泡咖啡粉");
    }
    @Override
    protected void addCondiments() {
        System.out.println("加糖和牛奶");
    }
}

// 子类 B：茶
class Tea extends BeverageMaker {
    @Override
    protected void brew() {
        System.out.println("用沸水浸泡茶叶");
    }
    @Override
    protected void addCondiments() {
        System.out.println("加柠檬");
    }
}
```

**区别在于：**

- **真·模板方法**：父类说：“你先做A，再做B，最后做C。” 子类只负责实现 A、B、C 的具体内容。
- **你的 BaseApi**：父类说：“我有一个工具叫 `req()`，你拿去用吧。” 至于子类是用它来 `get` 还是 `post`，是先 `header` 还是先 `body`，父类完全不管。

**结论：** 你的 `BaseApi` 更准确的分类是：

1. **抽象基类（Abstract Base Class）**：提供共享状态（`factory字段`）和共享行为（`req方法`）。
2. **辅助类/工具父类（Helper Superclass）**：类似于 JDK 中的 `AbstractList` 或 `AbstractSet`，它们提供了一些默认实现，但不强制规定子类的业务流程。

> - **模板方法模式** = **父类定流程** + **子类填细节**。
> - **你的 BaseApi** = **父类给工具** + **子类自由发挥**。
>
> **它借鉴了模板模式中**复用骨架代码**的思想，但实际上是一个**提供共享基础设施的抽象基类

## 泛型

```
List list = new ArrayList();
list.add("hello");
list.add(123);          // ← 任何类型都能塞
list.add(new User());

String s = (String) list.get(0);   // 取出来要强转
```

### 不用泛型得问题:

- **编译器不检查类型**,不强转接收运行时容易爆 `ClassCastException`
- 取数据时**到处强转**,丑陋
- 看代码不知道里面装的什么

```java
List<String> list = new ArrayList<>();
list.add("hello");
list.add(123);          // ← 编译器报错,123 不是 String

String s = list.get(0);  // 不用强转,编译器知道是 String
```

`<String>` 这个就是**泛型参数**,告诉编译器"这个 List 装的是 String"。

### 泛型的好处

1. **编译期检查**:类型不对编译就报错,不等运行时崩
2. **不用强转**:取出来直接是正确类型
3. **代码可读**:`List<User>` 一眼知道装的什么

**一个 CommonResp 类,装得下任意类型的 data**。这就是泛型的威力。

> 顺带一提:`T` 是约定俗成的字母,意思 **T**ype。其他常用占位符:
>
> - `E` = **E**lement(元素,集合常用)
> - `K, V` = **K**ey, **V**alue(Map 常用)
> - `R` = **R**eturn(返回值)
> - `?` = 通配符,代表"任意类型"

### 什么是类型擦除

早期为了让有泛型的代码和没泛型的代码**能一起跑**，代价就是:**编译时检查泛型,但编译完成后泛型信息就被"擦除"了**。

```java
List<String> stringList = new ArrayList<>();
List<Integer> intList = new ArrayList<>();

System.out.println(stringList.getClass() == intList.getClass()); 
//相当于ArrayList == ArrayList
// 输出:true ⚠️
```

**两个类型不同的 List,运行时是同一个类**。

因为编译完之后,**泛型信息被擦除了**:

```java
// 你写的代码(编译前)
List<String> list = new ArrayList<>();
list.add("hello");
String s = list.get(0);

// 编译后实际变成(等价于)
List list = new ArrayList();
list.add("hello");
String s = (String) list.get(0);   // 编译器自动加的强转
```

**泛型只在编译期帮你检查类型,运行时其实就是个普通的 List**。

### 类型擦除的几个证据

```java
List<String> list = new ArrayList<>();

// 证据 1:运行时拿不到泛型
list.getClass();  // ArrayList,不是 ArrayList<String>

// 证据 2:不能 instanceof 泛型
if (list instanceof List<String>) { }  // ❌ 编译错误

// 证据 3:不能 new 泛型数组
T[] arr = new T[10];                    // ❌ 编译错误

// 证据 4:同一个类不能同时有 List<String> 和 List<Integer> 的方法
public void m(List<String> l) { }
public void m(List<Integer> l) { }     // ❌ 编译错误,擦除后两个方法签名一样
```

**一句话记忆**

> **Java 的泛型是"伪泛型"——编译时严格,运行时擦除。** 编译器知道,JVM 不知道。

### 类型擦除给反序列化带来的问题

1. 序列化 (Serialization)

   对象 (内存) → 字节流/文本 (存储或传输)

   - 是把内存中活生生的 Java 对象（比如 `new SysTokenVO()`）变成一串可以保存或发送的数据（比如 JSON 字符串、二进制字节）。

2. 反序列化 (Deserialization)

   字节流/文本 (存储或传输) → 对象 (内存)

   - 是把那串数据（比如 JSON 字符串）变回内存中可以操作的 Java 对象。

#### Jackson 反序列化是怎么工作的

`response.as(SomeClass.class)` 内部用 Jackson(或类似库)做反序列化:

```
JSON 字符串 → Jackson 看你给的 Class → 创建对象 → 填字段
```

Jackson 拿到 `CommonResp.class`,它知道:

- 字段 `code`(Integer)
- 字段 `msg`(String)
- 字段 `data`...

**data 是什么类型?Jackson 不知道**——因为 CommonResp 类本身的 data 字段类型是 `T`,T 已经被擦除了。

Jackson 处理不知道类型的字段,会怎么做?**默认转成 LinkedHashMap**(就是把 JSON 对象当成 Map):

```java
CommonResp<SysTokenVO> resp = response.as(CommonResp.class);
SysTokenVO token = resp.getData();   // ⚠️ 运行时报 ClassCastException
//   ↑ 实际是 LinkedHashMap,强转 SysTokenVO 失败
```

编译器以为 `getData()` 返回 SysTokenVO(因为泛型),**运行时其实是 LinkedHashMap**——编译期和运行期对类型的认知**对不上**。

### TypeRef 怎么解决这个问题

```java
CommonResp<SysTokenVO> resp = response.as(new TypeRef<CommonResp<SysTokenVO>>() {});
//                                          ↑ 注意这后面有个 {} ,不是普通的 new
```

注意 `new TypeRef<...>() {}` 后面那个 **{}**——这不是空 body,是**创建了 TypeRef 的匿名子类**。

#### 这一招的精妙之处

虽然泛型在运行时被擦除,但**有一种情况下泛型信息会被保留**:

> **类的"父类的泛型参数",会保留在类的元数据里**。

也就是说,**子类知道自己的父类用了什么泛型**。

### 怎么识别是匿名子类？

匿名内部类和匿名本质上是一个东西。区别在于，匿名子类要

- `new Thread() {}` -> 父类是 `Thread`。
- `new Runnable() {}` -> 父类是 `Object`，实现了 `Runnable` 接口。

匿名内部类常用于**事件监听**、**回调函数**、**线程启动**
匿名子类常用于**模板方法模式**、**泛型类型捕获（如 TypeRef）**

当你看到 `new XXX() { ... }` 时，看 `XXX` 是什么：

### 怎么看“父子关系”？（判断技巧）

| `XXX` 的类型                        | 语法动作     | 父子关系描述                | 例子                                           |
| :---------------------------------- | :----------- | :-------------------------- | :--------------------------------------------- |
| **普通类** (如 `Thread`, `TypeRef`) | `extends`    | 匿名类是 `XXX` 的**子类**   | `new Thread() {}` 👉 是 Thread 的子类           |
| **抽象类** (如 `HttpServlet`)       | `extends`    | 匿名类是 `XXX` 的**子类**   | `new HttpServlet() {}` 👉 是 HttpServlet 的子类 |
| **接口** (如 `Runnable`, `List`)    | `implements` | 匿名类是 `XXX` 的**实现类** | `new Runnable() {}` 👉 实现了 Runnable 接口     |

```java
new TypeRef<CommonResp<SysTokenVO>>() {}
```

- `TypeRef` 是一个**抽象类**（Abstract Class）。
- 所以，这个匿名类 **extends TypeRef**。
- 结论：它是 `TypeRef` 的**匿名子类**。

### 不用 TypeRef 还有别的办法吗

有,几种常见替代方案:

#### 方案 1:不用泛型壳,直接展开(粗暴但能跑)

```java
// 别建 CommonResp<T>,直接为每种响应建独立类
public class LoginResp {
    private Integer code;
    private String msg;
    private SysTokenVO data;
}

LoginResp resp = response.as(LoginResp.class);    // 这就行,不用 TypeRef
```

代价:**每个接口都要重复 code/msg 字段**,失去了泛型壳的复用价值。

#### 方案 2:用 path 取值(简单场景)

```java
String token = response.path("data.access_token");
Integer code = response.path("code");
```

不需要 POJO,**简单但失去 IDE 提示和编译期检查**。异常用例可以用,正向用例不推荐。

#### 方案 3:用 Jackson 的 TypeReference(等价物)

```java
ObjectMapper mapper = new ObjectMapper();
CommonResp<SysTokenVO> resp = mapper.readValue(
    response.asString(), 
    new TypeReference<CommonResp<SysTokenVO>>() {}    // ← Jackson 的 API
);
```

`TypeReference` 是 Jackson 的版本,`TypeRef` 是 RestAssured 包装过的。**原理一样**。

#### 推荐:用 TypeRef

最简洁,语义清晰,RestAssured 体系内一致。**记住这个写法就行**。

### 对应的源码理解(简化版)

RestAssured 的 `as(TypeRef)` 大概是这样实现的(简化):

```java
// 在 Response 实现类里(比如 RestAssuredResponseImpl)
public <T> T as(TypeRef<T> typeRef) {
    // 1. 从 TypeRef 拿到完整类型(包括泛型参数)
    Type type = typeRef.getType();    // ← 关键:从父类泛型读出来的
    
    // 2. 拿响应字符串
    String body = this.asString();
    
    // 3. 转交给 Jackson(或其他反序列化器)
    ObjectMapper mapper = ...;
    return mapper.readValue(body, mapper.constructType(type));
}
```

而 TypeRef 本身大概长这样:

```java
public abstract class TypeRef<T> {
    private final Type type;
    
    protected TypeRef() {
        // 构造时,通过反射读取自己父类的泛型参数
        Type superclass = getClass().getGenericSuperclass();
        // ↑ getClass() 是匿名子类,getGenericSuperclass() 拿到 TypeRef<CommonResp<SysTokenVO>>
        this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
        // ↑ 提取出 CommonResp<SysTokenVO>,存到 type 字段
    }
    
    public Type getType() {
        return type;
    }
}
```

**关键**:TypeRef 的构造函数里**用反射读取自己父类的泛型**——这个动作发生在 `new TypeRef<...>(){}` 那一刻。

## 为什么 Api 里不该写 `.then()`?

```
职责分离:
Api 类的职责: 发请求 → 返回原始响应
测试类的职责: 拿到响应 → 做断言
```

```java
public Response save(String token, String body) {
    return req(token).body(body)
            .when().post("/sys/user")
            .then().extract().response();        // 冗余
}
```

Api 里 `.then().extract().response()` 没接任何校验项,等于绕了一圈把 Response 拿回来。功能上无害,但是潜在风险——后人可能在中间加上真校验,污染 Api 的职责边界。

### 为什么不测正向msg文案

正向响应的 msg 在大多数系统里**就是个固定占位符**:

- Maku:`"msg": "success"`
- Spring 老项目:`"msg": "操作成功"`
- 阿里规范:`"message": "OK"`

**没有业务含义**,测了也只是测"后端没改这个字符串"——价值近乎 0。

如果哪天后端把 success 改成 ok,你的测试全挂——但**业务没出问题**。这就是"脆弱测试"——失败和真实问题没关系。

 测 data:
  - 简单接口(data=null):跳过,改用 DB 验证业务效果
  - 业务接口(data 有结构):用 JSON Schema 校验结构 + 关键字段值

### 为什么异常响应必测msg

异常 msg 是**用户体验的关键路径**:

```
用户填表 → 后端校验失败 → 返回 msg → 前端展示给用户
                              ↑
                         "用户名不能为空" 这条信息
                         如果文案错了,用户看到乱码或错信息
```

这个文案是产品功能的一部分,需要测试保护:

- 后端误把"用户名不能为空"改成"用户名 not null":bug
- 后端去掉了这条校验:bug(校验不生效)
- 后端把"用户名不能为空"和"邮箱不能为空"互换了:bug

### 为什么异常的 data 不测

异常时 data 通常是 null/空串/不存在,Maku 这种是 `"data": null`。

- 测 `data == null` 价值不大,跟测 msg 没关系
- 真要测就用 `assertThat(resp.path("data")).isNull()`,但意义不大

**异常用例只测 code + msg 就够**。



## 主流公司接口文档

#### 大厂 / 头部公司(阿里、腾讯、字节、美团等)

- **工具自动生成 + 开发手动补全**
- 后端用 Swagger / SpringDoc 注解,启动后自动生成 OpenAPI JSON
- 大公司会有**专门的 API 平台**(类似 YApi、Apifox 内部版),开发提交后自动同步,测试在平台上看
- **响应示例、错误码、msg 文案**会强制要求开发标注
- 前后端联调前,开发要先把文档写完整(否则前端不开工)

#### 中型公司(50-500 人)

- **工具生成 + 开发偶尔补充**
- 用 Swagger 自动生成,但开发不爱写注释
- 文档**字段类型有,响应示例残缺,错误情况不写**(就是你看 Maku 这样)
- 测试经常要**自己跑接口拿响应** → 反推文档

#### 中小公司 / 创业公司

- **靠测试自己摸**
- 接口文档?后端嘴里说一下,Postman 里有几个示例
- 错误码:口头沟通"出错就给你 -1"
- 接口字段:**直接看代码或问开发**

**结论**:**80% 的接口文档不完整,这是常态,不是异常**。

## 接口文档不全怎么补？

第一步：抓包手动整理接口文档。跑下正向用例，再手动跑下异常用例，手动整理记录。

第二步：如果测试人员有后端代码权限，翻后端代码核对。如果没权限,**找最熟悉这个模块的开发问 5 分钟**。

第三步：找开发或产品核对接口。

实际上在做接口自动化时除了跑正向，需要将异常用例跑一下，记录一下响应。

```
正向场景跑一遍 → 记录:
- HTTP 状态码
- 响应结构(JSON 长啥样)
- 成功 msg 文案

异常场景一个个试 → 记录:
- 不传必填字段
- 字段超长
- 字段格式错(邮箱不是邮箱格式)
- 业务冲突(用户名重复)
- 未授权
- 无效 token
- 越权访问
```

比如说，跑POST /sys/user正向用例，正向跑一次，然后再跑重复用户名的异常用例，获取输出。响应:

```
{"code":500,"msg":"用户名已经存在","data":null}
```

测试代码中补充断言：

```java
assertThat((Integer) response.path("code")).isEqualTo(500);  // ← 补上具体值
assertThat((String) response.path("msg")).isEqualTo("用户名已经存在");  // ← 补上文案
```

## 看不到后端代码做接口测试的经验

#### 经验 1:Spring Boot 项目几乎都用同一套校验

```
看到 Spring Boot → 心里默认:必填校验 = @NotNull/@NotBlank
                     格式校验 = @Pattern/@Email
                     长度校验 = @Size

所以:测一个必填字段就代表测了校验机制
```

**这个"心理默认"省了你大量沟通时间**。前提是项目用主流栈,不是某个奇葩自研框架。

#### 经验 2:CRUD 接口的"套路"是固定的

```
看到 POST /xxx → 套用"新增类用例模板"(7 个关注点)
看到 PUT /xxx  → 套用"修改类用例模板"(6 个关注点)
看到 DELETE   → 套用"删除类用例模板"(5 个关注点)
```

**记住套路,看到接口直接套**——这就是接口测试人员"快"的秘密。

#### 经验 3:同类型字段的反应高度相似

```
后端的 String 必填校验,99% 用 @NotBlank
后端的 Long 必填校验, 99% 用 @NotNull  
后端的 Integer 校验,  通常额外有 @Min/@Max

所以:测 username 缺失,基本等同于测了所有 String 必填
```

#### 经验 4:Maku 的 msg 文案有套路

跑一次 Maku 的接口你就发现:

```
"xxx不能为空"   → 必填校验
"参数错误"      → 格式校验
"还未授权,不能访问"  → 权限校验
"系统错误"      → 服务端异常
```

**学会了一个接口的错误模式,其他接口大概率一样**。这又省你时间。

### 流水线设计原则

- **Fail Fast**:快速失败,提早暴露问题
- **可重复**:任何人在任何机器上跑结果一致
- **可观测**:每一步都有日志和指标
- **幂等性**:重复跑同样的输入,结果一样
- **小步快走**:小改动小流水线,而不是憋大版本
- **分支隔离**:特性分支不影响主干

### 设计请求规格工厂

如果面试问“你怎么设计请求规格工厂的，为什么不用static”

> v3 把请求规格工厂做成了实例对象而不是 static 工具类。一是 baseSpec 创建一次复用,避免每次发请求都重建 Builder 和 Filter,在大批量用例下能减少 GC 压力。二是 Env 通过构造函数注入,工厂可以被 Spring 管理,也方便我们对接多套环境的回归。第三是 static 字段在多线程下风险高,实例字段配合 final 修饰天然线程安全。



### 设计测试基类

如果面试问"你怎么设计测试基类的",你可以这么讲(覆盖 6 个知识点):

> "我设计了一个抽象 BaseTest 类,用 `@TestInstance(PER_CLASS)` 让 JUnit 把测试类实例的生命周期变成类级别,这样 token、JdbcTemplate 这些可以做成实例字段而不是静态字段——避免了多测试类并发时静态变量被互相覆盖的问题。

>
> 父类的字段都用 protected 暴露给子类,Api 层通过 RequestSpecFactory 构造,所有依赖通过实例传递,不读全局状态——这套结构跑 JUnit5 并行配置时是天然安全的。"

### 超令牌解决泛型擦除导致反序列化时读取类型不对问题

>"Java 泛型有个很关键的特性叫类型擦除——编译时编译器会用泛型做严格的类型检查,但编译完后泛型信息会被擦除掉,运行时一个 List<String> 和 List<Integer> 是同一个 ArrayList。
>
>这个特性会给反序列化带来问题。比如我们框架里的通用响应壳 `CommonResp<T>`,如果直接 `response.as(CommonResp.class)`,Jackson 拿到 Class 对象时,T 这个类型参数早就被擦除了——它不知道 data 字段该反序列化成什么,默认会转成 LinkedHashMap,后续按 SysTokenVO 取值就 ClassCastException。
>
>解决办法是用 TypeRef,核心 trick 是创建匿名子类——`new TypeRef<CommonResp<SysTokenVO>>() {}`,后面的花括号是关键。Java 类型擦除有个例外:**子类的父类泛型参数,会被保留在字节码里**。TypeRef 内部用反射读取这个父类泛型,就能拿到完整的 `CommonResp<SysTokenVO>` 类型信息,Jackson 才知道 data 是 SysTokenVO。
>
>这个手法叫'超类型令牌'(Super Type Token),Jackson 自己有 TypeReference,Spring 有 ParameterizedTypeReference,都是同一个原理。"



### 配置文件防御构建物文件缺失导致读错环境编程

> "为提高框架健壮性,设计多重防御机制:配置加载采用 Fail Fast 模式,缺失立即报错;数据库连接做白名单校验,拒绝连接非测试环境;启动自检测试覆盖关键资源完整性。该机制避免了至少一次因 CI 构建配置变更导致的环境误连接事故。"

### 异常响应粒度

如果面试问"你怎么设计响应断言粒度",可以这么答:

> "我对正向和异常响应做不同处理。
>
> **正向**只测 code=0 和 data 结构(用 JSON Schema 校验),不测 msg——因为 success 这种文案在大多数系统里是占位符,测它脆弱性高,业务价值低。增删改的业务效果走 DB 验证。
>
> **异常**必测 code 非 0 和 msg 精确文案——因为提示信息是用户能感知的功能点,需要测试保护。比如把"用户名不能为空"改成"用户名 not null"是 bug,我的测试要能拦下。
>
> 这种分层让测试既严谨又不脆弱——后端改了无关紧要的成功提示语,我的测试不挂;真的改坏了用户能看到的错误提示,我的测试一定挂。"

### 设计接口测试用例

被问"你怎么设计接口测试用例?",可以这么答:

> "我有一套自己的工作流,大概是这样:
>
> **第一步分类**:看接口是 CRUD 哪一类,套对应的用例模板。比如新增类我会关注必填校验、唯一约束、关联存在性（设置不存在的关联数据）、数据持久化（数据有没有在数据库）这几个点。
>
> **第二步定边界**:用等价类划分选代表,不全测。比如必填字段我选 2-3 个不同类型的代表测,因为后端通常用 Spring Validation 统一校验,测一个相当于验证了机制。

> **第三步看边界**:边界值会单独多花时间——下界、上界、上界+1,因为开发写 if 判断符号容易错。

> **第四步推测**:基于经验补几个'容易出 bug 的点',比如空字符串、特殊字符、并发调用。

> **第五步落地**:Postman 先跑一遍看实际行为,再写代码——避免凭文档臆测。

>
> 一套配置全覆盖。"

### 环境管理

被问"环境配置怎么管理":

> "我用 Spring Profile + 环境变量两层结构。
>
> profile 文件管**结构性配置**(数据库 URL、连接池、日志级别), 这部分进代码仓库,可追溯、可 review。
>
> 环境变量管**敏感和易变配置**(密码、密钥、域名), 通过 .env 文件或者 GitHub Secrets 注入,不进仓库。
>
> 我项目里 application.yml 默认用 dev,本地开发零配置。 application-prod.yml 在 deploy/ 目录,CI 部署时由 SCP 上传到服务器。 服务器 docker-compose 启动时设 SPRING_PROFILES_ACTIVE=prod,自动加载。
>
> 这个分层是行业主流做法,大厂会再上配置中心(Apollo / Nacos), K8s 环境会用 ConfigMap + Secret,但核心思想都一样: **结构进代码,密钥进环境**。"

### 打包

被问"Spring Boot 怎么打包的":

> "Spring Boot 用 spring-boot-maven-plugin 打成 fat jar——把所有依赖、 配置、自己的代码全部打进一个 jar。结构上 BOOT-INF/classes 放业务代码, BOOT-INF/lib 放依赖 jar(直接放 jar 而不是解压,这是 Spring Boot 自己 的特殊 classloader 实现的)。
>
> 这种打包方式让部署变得极简——传一个 jar,java -jar 直接跑, 不用维护 classpath。"

### 多模块 vs 微服务

被问"你做的 Maku 是不是微服务":

> "不是。Maku 是 Spring Boot 单体 + Maven 多模块的架构。多模块是代码 组织手段——maku-framework / maku-module-system 等模块各管一块, 编译时分开维护,但 maku-server 模块依赖了它们,打包时把所有模块的代码 合成一个 fat jar,运行时是一个进程。
>
> 这和微服务架构完全不同。微服务是每个业务模块独立进程、独立数据库、 通过网络通信。Maku 这种结构适合中小项目——保留代码清晰度,但避免 了分布式系统的复杂度。"
