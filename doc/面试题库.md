# Maku 接口自动化测试框架 — 面试题库

---

## 一、框架设计类

### Q1：请介绍一下你的接口自动化框架是怎么设计的？

> 我的框架分四层：
>
> 配置层（Env）负责环境配置加载，
>
> 请求规格工厂（RequestSpecFactory）统一管理 HTTP 请求的公共配置（baseUri、ContentType、Filter、超时），
>
> 业务 API 层封装各模块的接口路径和参数并返回原始 Response
>
> 测试类只做调用和断言。
>
> 核心设计原则是每层只做一件事、不越权。测试类不知道 URL 怎么拼，API 类不做断言，RequestSpecFactory 不关心业务。这样改动一层不影响其他层。

### Q2：为什么用 RequestSpecFactory 而不是直接用 RestAssured 全局配置？

> v1 的做法是在 BaseTest 里直接 `RestAssured.baseURI = "..."` 修改全局静态变量。多测试类并发时，类 A 的 setAll() 会覆盖类 B 的配置，导致请求发到错误路径或 token 丢失，出现 401/404。
>
> v3 把请求规格做成了实例对象。RequestSpecFactory 内部用 RequestSpecBuilder 构建 baseSpec 快照，每次调用 unauth()/auth() 都通过 `given().spec(baseSpec)` 返回新实例，互不干扰。而且 baseSpec 是 `private final` 的，JMM 保证构造函数结束后其他线程能立刻看到最终值，天然线程安全。

### Q3：为什么不用 static 工具类而用实例化的工厂？

> 三个原因。一是 baseSpec 创建一次复用，避免每次发请求都重建 Builder 和 Filter，在大批量用例下能减少 GC 压力。二是 Env 通过构造函数注入，工厂可以被 Spring 管理，也方便对接多套环境的回归。三是 static 字段在多线程下风险高，实例字段配合 final 修饰天然线程安全。

### Q4：你怎么设计测试基类的？

> 我设计了一个抽象 BaseTest 类，用 `@TestInstance(PER_CLASS)` 让 JUnit 把测试类实例的生命周期变成类级别，这样 token、JdbcTemplate 这些可以做成实例字段而不是静态字段，避免了多测试类并发时静态变量被互相覆盖的问题。
>
> 公共流程比如登录、加载环境配置、创建请求工厂放在 `@BeforeAll` 里。数据清理用模板方法模式，父类定义钩子，子类按自己的测试标识精确清理，避免 v1 那种全表 delete 互相伤害的问题。
>
> 父类的字段都用 protected 暴露给子类，Api 层通过 RequestSpecFactory 构造，所有依赖通过实例传递，不读全局状态。这套结构跑 JUnit5 并行配置时是天然安全的。

### Q5：你的请求模板机制是怎么设计的？为什么不用 POJO 构造请求？

> 请求侧统一用 JsonTemplateUtil + JSON 模板，不建请求 POJO。原因是异常用例需要灵活删字段（必填项缺失）或改字段（错误值），模板的 `.set()` / `.delete()` 比新建一个 POJO 灵活得多。
>
> JsonTemplateUtil 封装了 JsonPath，提供流式 API：`load()` 加载模板、`set()` 改值、`delete()` 删字段、`toJson()` / `toMap()` 输出。关键约束是 POST/PUT 用 `toJson()` 输出 JSON 字符串，GET 用 `toMap()` 输出 Map 给 queryParams，两者序列化行为不同，混用会出问题。
>
> 响应侧按需建 POJO，用 CommonResp<T> 统一响应壳，分页用 PageResult<T>，避免每个接口重复 code/msg 字段。

## 二、并发安全类

### Q6：你的框架怎么保证并发安全的？

> 核心思路是"能不共享就不共享"，而不是"共享了加锁"。
>
> 具体做了三点：第一，token 从 static 字段改成实例字段，配合 @TestInstance(PER_CLASS)，每个测试类有自己的 token，互不影响。第二，RestAssured 的全局静态配置（baseURI、basePath）不再直接修改，改用 RequestSpecFactory 的实例化 baseSpec，每次请求基于模板新建 spec。第三，JSON 模板对象不再是类共享字段，改为方法内局部变量，链式调用后立即消费，不存在跨方法存活的问题。
>
> 局部变量在线程栈上，天然线程安全。这是比加锁更简单更稳的方案。

### Q7：v1 框架的 static token 有什么问题？详细解释一下。

> `public static String token` 意味着这个字段在 JVM 方法区只有一份，所有测试线程读写的是同一个内存地址。单线程跑没问题，但开 JUnit 并行后，类 A 登录拿到 tokenA 写入字段，类 B 登录拿到 tokenB 覆盖了它，类 A 再用 token 就变成了 tokenB。如果 tokenB 是另一个账号的权限，用例会因权限不足报 401。更危险的是异常用例可能把 token 设空，其他类拿空 token 调接口全部失败。
>
> static 字段 + 多线程写 = 竞态条件。解决不是加锁，而是让每个线程有自己的副本——实例字段。

### Q8：DocumentContext 作为共享字段有什么问题？

> DocumentContext 是 JsonPath 的核心类，它是可变对象——`.set()` / `.delete()` 改的是它本身，不是返回新对象。如果把它作为父类的实例字段，类级别并发下没问题（同一个类的方法还是串行的），但方法级别并发下，多个测试方法同时改同一个 DocumentContext 实例，数据会乱。
>
> v3 的做法是把 DocumentContext 从字段下沉到方法内的局部变量，每次 `JsonTemplateUtil.load()` 新建一个，链式调用后立即 `toMap()` / `toJson()` 消费掉，不跨方法存活。局部变量在线程栈上，天然隔离。

### Q9：JVM 内存模型中，哪些区域是线程安全的？

> 栈是每线程独享的，放局部变量，天然线程安全。堆是所有线程共享的，放 new 出来的对象。方法区放类信息和 static 字段，全 JVM 一份。
>
> 所以 static 非 final 字段 + 多线程 = 大概率有 bug。final 字段无论 static 与否都是安全的，因为 JMM 保证构造函数结束后其他线程能立刻看到最终值。
>
> 解决并发的优先级：不共享（最简单）> 不可变（共享但只读）> 加锁同步（能用但容易写错）。

## 三、泛型与反序列化类

### Q10：Java 泛型擦除是什么？给接口测试带来了什么问题？

> Java 的泛型是伪泛型——编译时编译器用泛型做严格的类型检查，但编译完后泛型信息被擦除了。运行时 `List<String>` 和 `List<Integer>` 是同一个 ArrayList。
>
> 这给反序列化带来了问题。比如我们的 CommonResp<T>，如果直接 `response.as(CommonResp.class)`，Jackson 拿到 Class 对象时 T 已经被擦除，不知道 data 字段该反序列化成什么类型，默认转成 LinkedHashMap。后续按 SysTokenVO 取值就会 ClassCastException。

### Q11：TypeRef 是怎么解决泛型擦除的？原理是什么？

> TypeRef 利用了类型擦除的一个例外：**子类的父类泛型参数会被保留在字节码里**。
>
> `new TypeRef<CommonResp<SysTokenVO>>() {}` 后面的花括号创建了 TypeRef 的匿名子类。TypeRef 的构造函数里用反射读取自己父类的泛型参数——`getClass().getGenericSuperclass()` 拿到 `TypeRef<CommonResp<SysTokenVO>>`，然后从中提取出完整的 `CommonResp<SysTokenVO>` 类型信息。Jackson 拿到完整类型后就能正确反序列化 data 字段。
>
> 这个手法叫"超类型令牌"（Super Type Token），Jackson 的 TypeReference、Spring 的 ParameterizedTypeReference 都是同一个原理。

### Q12：不用 TypeRef 还有别的办法吗？

> 有三种替代方案，但各有代价：
>
> 第一种：不为 data 建泛型壳，每个接口建独立的 LoginResp、UserResp 类，直接 `response.as(LoginResp.class)`。代价是每个接口都要重复 code/msg 字段。
>
> 第二种：用 GPath 直接取值，`response.path("data.access_token")`。不需要 POJO，但失去 IDE 提示和编译期检查。异常用例可以用，正向不推荐。
>
> 第三种：用 Jackson 的 TypeReference，原理和 TypeRef 一样，只是不同库的 API。
>
> 推荐 TypeRef，最简洁，RestAssured 体系内一致。

## 四、配置与环境类

### Q13：环境配置怎么管理的？为什么优先级是 -D > 环境变量 > 配置文件 > 默认值？

> 我用 properties 文件 + -Denv 切换环境。配置加载遵循四级回退：系统属性 > 环境变量 > properties 文件 > 默认值。
>
> 这个顺序基于"谁离用户最近，谁说了算"。-D 是用户的即时意图，优先级最高。环境变量代表当前运行环境的真实身份，由运维或 CI 注入，敏感信息不进代码仓库。配置文件是团队共享的开发默认值。硬编码默认值是最后兜底。
>
> 这样做的好处是本地开发用配置文件零配置，CI 通过环境变量注入密码不泄露，临时调试用 -D 不用改文件。

### Q14：什么是饿汉式单例？为什么用饿汉式不用懒汉式？

> 饿汉式是类加载时立即初始化单例实例。对比懒汉式是第一次调用时才初始化。
>
> 饿汉式利用 JVM 的类加载机制保证线程安全——JVM 保证一个类的 static 字段初始化是线程安全的且只执行一次。懒汉式在多线程下要写双重检查锁定 + volatile，复杂、易错、性能差。
>
> 行业经验：除非单例创建特别贵（比如要 30 秒），否则永远用饿汉式。这是《Effective Java》明确推荐的。

### Q15：配置文件找不到时怎么处理？

> 当前实现是找不到配置文件时静默降级到全默认值，方便 IDE 首次运行不报错。但这是个潜在风险——如果 CI 的 maven-resources-plugin 配置错误，config 目录没被复制到 target，代码不会报错，但实际连的是默认环境而不是目标环境。
>
> 改进方向是 Fail Fast：配置文件缺失时默认抛异常，保留 `-Denv.allowMissing=true` 作为新人逃生口。防止 CI 构建配置变更导致的环境误连接。

## 五、断言与用例设计类

### Q16：你怎么设计响应断言粒度的？

> 我对正向和异常响应做不同处理。
>
> **正向**只测 code=0 和 data 结构（用 JSON Schema 校验），不测 msg——因为 success 这种文案在大多数系统里是占位符，测它脆弱性高，业务价值低。如果后端把 success 改成 ok，测试挂了但业务没出问题，这就是"脆弱测试"。增删改的业务效果走 DB 验证。
>
> **异常**必测 code 非 0 和 msg 精确文案——因为提示信息是用户能感知的功能点，需要测试保护。比如把"用户名不能为空"改成"用户名 not null"是 bug，我的测试要能拦下。
>
> 这种分层让测试既严谨又不脆弱——后端改了无关紧要的成功提示语，我的测试不挂；真的改坏了用户能看到的错误提示，我的测试一定挂。

### Q17：你怎么设计接口测试用例？

> 我有一套自己的工作流：
>
> **第一步分类**：看接口是 CRUD 哪一类，套对应的用例模板。比如新增类我会关注必填校验、唯一约束、关联存在性、数据持久化这几个点。
>
> **第二步定边界**：用等价类划分选代表，不全测。比如必填字段我选 2-3 个不同类型的代表测，因为后端通常用 Spring Validation 统一校验，测一个相当于验证了机制。
>
> **第三步看边界**：边界值会单独多花时间——下界、上界、上界+1，因为开发写 if 判断符号容易错。
>
> **第四步推测**：基于经验补几个容易出 bug 的点，比如空字符串、特殊字符、并发调用。
>
> **第五步落地**：Postman 先跑一遍看实际行为，再写代码——避免凭文档臆测。

### Q18：接口文档不全怎么办？

> 接口文档永远不全，自己摸索是常态。我的做法：
>
> 第一步抓包手动整理——正向跑一遍记录响应结构，异常一个个试记录错误响应。
>
> 第二步有代码权限就翻后端代码核对，没权限就找最熟悉这个模块的开发问 5 分钟。
>
> 第三步找开发或产品核对接口。
>
> 几个经验：Spring Boot 项目几乎都用同一套校验（@NotNull/@NotBlank），CRUD 接口的套路是固定的，同类型字段的反应高度相似。学会一个接口的错误模式，其他接口大概率一样。

### Q19：为什么不测正向的 msg，但异常的 msg 必须测？

> 正向的 msg 在大多数系统里是固定占位符（success/OK/操作成功），没有业务含义。测了只是测"后端没改这个字符串"——价值近乎 0。如果后端把 success 改成 ok，测试全挂但业务没出问题，这是脆弱测试。
>
> 异常的 msg 是用户体验的关键路径——用户填表校验失败，后端返回 msg，前端展示给用户。如果"用户名不能为空"被改成"用户名 not null"，用户看到乱码，这是真实的 bug。所以异常 msg 必须精确断言保护。

### Q20：delete 和 set null 有什么区别？为什么测必填缺失用 delete？

> `.delete("$.username")` 让字段从 JSON 中完全消失：`{}`。
> `.set("$.username", null)` 保留字段但值为 null：`{"username": null}`。
>
> 在服务端，Spring 的 @NotBlank 校验的是"字段存在且非空字符串"，而字段不存在和字段为 null 可能走不同的校验逻辑。测"必填缺失"应该用 delete，模拟的是前端根本没传这个字段的场景。

## 六、数据驱动类

### Q21：CSV 数据驱动怎么组织的？为什么一个 CSV 只放一种逻辑？

> 一个 CSV 只放一种参数处理逻辑——invalid_xxx.csv 只放 set 改字段的场景，missing_field.csv 只放 delete 删字段的场景。
>
> 如果混在一个 CSV 里，测试方法要写 if-else 判断"这条是改字段还是删字段"，代码丑且易错。分开后每个测试方法内部无 if-else，逻辑单一清晰，加新场景只改 CSV 不改代码。
>
> 比如 missing_field.csv 的列是 description/missingField/expectedMsg，delete 的路径动态拼接：`delete("$." + missingField)`。CSV 加一行就能覆盖一个新的必填字段，测试方法本身不需要改。

### Q22：模板和 CSV 的分工是什么？

> 模板管"全集"，CSV 管"差异"。
>
> 模板按接口文档列全所有字段（包括可选字段），默认值跑通正向用例。这样模板是接口文档的可执行版本，缺字段等于遮蔽了接口能力。而且异常用例依赖 `.delete()` 构造"字段缺失"场景，模板里不存在的字段删不了。
>
> CSV 只列本次要变化的字段和期望结果，不变的字段由模板兜底。这避免了每行重复所有字段的冗余。

## 七、CI/CD 类

### Q23：你的 CI/CD 怎么做的？

> 我做了三层触发：
>
> **第一层是上游驱动**——被测系统代码 push 后，通过 GitHub Actions 构建 + 部署到服务器，然后用 repository_dispatch 事件跨仓库触发测试框架跑全量回归。这模拟真实公司"后端发版前回归测试"的场景。
>
> **第二层是下游保活**——测试代码自身 push 时跑测试 + 部署报告，防止手抖提交了挂的测试。
>
> **第三层是定时回归**——每天凌晨 2 点全量跑，模拟夜间回归，早上看报告就知道头一天服务有没有问题。
>
> 三种触发场景对应了真实工程里 CI 的三种核心需求——发版门禁、代码保护、持续监控。

### Q24：流水线设计有哪些原则？

> 六个原则：Fail Fast（快速失败，提早暴露问题）、可重复（任何人在任何机器上跑结果一致）、可观测（每一步都有日志和指标）、幂等性（重复跑同样的输入结果一样）、小步快走（小改动小流水线）、分支隔离（特性分支不影响主干）。

### Q25：Allure 报告怎么集成的？

> 通过 allure-rest-assured 的 AllureRestAssured Filter，每次请求后自动把 curl 命令和响应体写入 Allure 报告附件。测试类用 @Feature/@Story/@DisplayName 注解标注模块、场景和用例描述。
>
> CI 里用 `mvn clean test allure:report` 生成报告，通过 GitHub Actions 部署到 GitHub Pages，面试官可以直接在线看测试结果，不用 clone 项目本地跑。这也是我对工程协作的理解：让 review 你工作的人，代价越低越好。

## 八、数据库验证类

### Q26：数据库验证怎么做？为什么用 JdbcTemplate？

> 我用 Spring JdbcTemplate 封装 JDBC 样板代码，配合 DruidDataSource 连接池。JdbcTemplate 不需要像 MyBatis 那样写 XML 映射，对于测试场景里的简单查询和删除足够用。
>
> 增删改类接口必须做 DB 验证：save 后查 DB 确认新增记录存在且字段正确，update 后查 DB 确认目标字段已变化，delete 后查 DB 确认记录已不存在（软删除场景查 deleted=0）。
>
> 查询类和登录类接口不做 DB 验证，因为没有数据变更。

### Q27：测试数据清理怎么做？

> 按 username 或业务标识前缀精确删除，比如 `DELETE FROM sys_user WHERE username LIKE 'test_v3_save_%'`。
>
> v1 用的是 `DELETE FROM sys_user` 全表删除，并发时一个进程清了数据，另一个进程验证失败。精确清理保证只删自己创建的数据，不影响其他测试类。
>
> 清理时机：@AfterEach 里执行，确保每条用例跑完都清理，不留脏数据。共享基础设施（如测试机构）不在 @AfterEach 中删除。

## 九、综合与场景类

### Q28：你的框架最大的技术改进是什么？

> 从 v1 到 v3 最大的改进是消除了三类并发隐患：静态 token 共享、RestAssured 全局 basePath 写入、DB 全表清理。
>
> 核心思路是把所有"共享可变状态"转为"实例独占"或"局部变量"。token 从 static 改为实例字段，请求配置从全局静态改为工厂实例，模板对象从父类字段改为方法局部变量，DB 清理从全表删除改为精确匹配。整个框架不需要任何锁或同步机制，就能在 JUnit5 并行模式下安全运行。

### Q29：baseSpec 是 private final 的，final 在这里有什么好处？

> 两个好处。第一，语义上保证 baseSpec 只在构造函数中赋值一次，之后整个生命周期不可变，不会被意外修改。第二，JMM（Java 内存模型）保证 final 字段在构造函数结束后，其他线程能立刻看到其最终值。普通字段没有这个保证——可能一个线程构造完对象后，另一个线程读到的是未初始化的旧值。所以 final 字段在多线程读场景下是安全的。

### Q30：unauth() 为什么不能直接 return baseSpec？

> 两个原因。第一，类型不对——baseSpec 是 RequestSpecification 类型，直接返回后没法链式调用 `.body()` `.post()` 等 RestAssured 的请求方法，只有 `given()` 返回的才是"可发请求"的接口。第二，共享风险——如果直接返回 baseSpec，调用方 `.header(...)` 一下就改了模板本身，后续所有人都受影响。`given().spec(baseSpec)` 的含义是"基于模板新开一个，你随便改"，不污染原始模板。

### Q31：你怎么保证框架的健壮性？

> 配置加载设计了多重防御：四级回退保证任何环境下都能拿到配置，Fail Fast 机制防止配置缺失时静默连错环境。数据库连接做了快速失效——connectTimeout 5 秒、socketTimeout 10 秒、maxWait 10 秒，DB 不通时快速失败而不是死等到 CI 超时被强杀。
>
> 测试数据用前缀标记 + @AfterEach 精确清理，保证用例之间互不干扰。请求规格每次新建实例，保证并发安全。Allure 报告自动附加请求/响应，失败时不用翻控制台就能定位问题。

### Q32：如果面试官问"你了解过哪些测试基建"？

### Q33：简历上怎么写这个项目？

> 重构 v1 框架，识别并消除三类并发隐患：静态 token 共享、RestAssured 全局 basePath 写入、DB 全表清理。引入 RequestSpecification 工厂模式，token 从静态字段下沉为测试类实例字段，DB 清理改为按测试标识精确删除。
>
> 可选加强：加上 Spring Boot 外壳后，简历可以写"基于 Spring Boot 集成的接口自动化测试框架"，Spring Boot 关键词命中率极高。

### Q34：Maku 是微服务架构吗？

> 不是。Maku 是 Spring Boot 单体 + Maven 多模块的架构。多模块是代码组织手段——各模块编译时分开维护，但 maku-server 模块依赖了它们，打包时合成一个 fat jar，运行时是一个进程。
>
> 微服务是每个业务模块独立进程、独立数据库、通过网络通信。Maku 这种结构适合中小项目——保留代码清晰度，但避免了分布式系统的复杂度。

### Q35：Spring Boot 怎么打包的？

> Spring Boot 用 spring-boot-maven-plugin 打成 fat jar——把所有依赖、配置、自己的代码全部打进一个 jar。结构上 BOOT-INF/classes 放业务代码，BOOT-INF/lib 放依赖 jar（直接放 jar 而不是解压，Spring Boot 用自己的 ClassLoader 实现）。
>
> 这种打包方式让部署变得极简——传一个 jar，java -jar 直接跑，不用维护 classpath。
