# Maku 接口自动化测试框架 v3 — 知识点梳理

---

## 一、框架架构总览

### 1.1 分层架构

```
测试类 (tests/)          →  调用 API + 断言，不关心请求细节
   ↓
业务 API 层 (api/)        →  封装接口路径和参数，return Response
   ↓
请求规格工厂 (client/)    →  统一管理 baseUri/Header/Filter/超时
   ↓
配置层 (config/)          →  读取环境配置，支持 -Denv 切换
```

**核心原则**：每层只做一件事，不越权。

- 测试类不知道 URL、Header 怎么拼
- API 类不做断言
- RequestSpecFactory 不关心业务
- Env 不关心 HTTP

### 1.2 目录结构（严格遵守）

```
src/
├── main/java/com/maku/apitest/
│   ├── config/        # Env 配置加载
│   ├── client/        # RequestSpecFactory（全局过滤器、鉴权、超时、日志）
│   ├── api/           # 业务模块 API 类（AuthApi、UserApi、OrgApi...）
│   ├── model/         # 响应 POJO
│   │   ├── common/    # CommonResp<T> / PageResult<T>
│   │   ├── auth/      # SysTokenVO
│   │   └── user/      # SysUserVO
│   └── utils/         # JsonTemplateUtil
└── test/
    ├── java/com/maku/apitest/tests/
    │   ├── base/      # BaseTest
    │   ├── auth/      # 一个接口一个测试类
    │   └── user/
    └── resources/
        ├── config/    # dev.properties / test.properties
        ├── template/  # JSON 请求模板
        ├── params/    # CSV 数据驱动
        ├── schemas/   # JSON Schema
        └── allure.properties
```

| 类别 | 技术选型 |
|---|---|
| 构建 | Maven |
| JDK | 17 |

## 二、核心组件详解

### 2.1 Env — 配置加载

**解决的问题**：同一套代码在不同环境（dev/test/prod）自动读取不同配置，不需要改代码。

**关键知识点**：

#### 饿汉式单例

```java
private static final Env INSTANCE = load();
public static Env get() { return INSTANCE; }
```

- 类加载时立即初始化，JVM 保证线程安全（ClassLoader 内部加锁）
- 对比懒汉式需要双重检查锁定 + volatile，饿汉式简单且安全
- 行业经验：除非单例创建特别贵，否则永远用饿汉式（《Effective Java》推荐）

#### 四级回退优先级

```
系统属性 (-Dkey=val) > 环境变量 (KEY) > properties 文件 > 默认值
```

| 优先级 | 来源 | 适用场景 |
|---|---|---|
| ① -D | 启动命令行 | 临时调试，优先级最高 |
| ② 环境变量 | CI/容器 | 敏感信息，不进代码仓库 |
| ③ properties | 项目配置文件 | 团队共享的环境配置 |
| ④ 默认值 | 代码硬编码 | 兜底，IDE 直接跑不报错 |

核心原则：**谁离用户最近，谁说了算**。临时指令 > 部署环境 > 项目默认 > 兜底。

#### try-with-resources

```java
try (InputStream is = Env.class.getClassLoader().getResourceAsStream(classpathPath)) {
    props.load(new InputStreamReader(is, StandardCharsets.UTF_8));
}
```

自动关闭流，等价于 finally 里写关闭逻辑。只要见到 InputStream/Connection/Statement 一律用此语法。

#### ClassLoader vs new File()

- `ClassLoader.getResourceAsStream()` 在 Maven 构建和 IDE 运行时都能正确找到 resources
- `new File("src/test/resources/...")` 依赖当前工作目录，CI 里经常出问题

### 2.2 RequestSpecFactory — 请求规格工厂

**解决的问题**：v1 直接修改 `RestAssured.baseURI` 等全局静态变量，多测试类并发时互相覆盖导致 401/404。

**关键知识点**：

#### 工厂模式 + 建造者模式

```java
this.baseSpec = new RequestSpecBuilder()     // 建造者：链式装配
        .setBaseUri(env.getBaseUrl())
        .setPort(env.getPort())
        .setContentType(ContentType.JSON)
        .addFilter(new AllureRestAssured())
        .build();                            // 收口产出不可变对象
```

- **工厂**回答"谁来造"（创建入口）
- **建造者**回答"怎么造"（组装步骤）
- `build()` 后的对象是快照，后续修改不影响它，可安全复用

#### 并发安全设计

每个线程有自己的：
- RequestSpecification 实例（`given().spec(baseSpec)` 每次返回新实例）
- token（通过参数传入，不读全局）
- basePath（写在自己的 spec 里）

#### private final baseSpec

- `private`：外部拿不到，保证模板不被弄脏
- `final`：字段只能赋值一次，生命周期内不变
- JMM 保证：构造函数结束后，其他线程能立刻看到 final 字段的最终值（普通字段没这个保证）

#### baseSpec 为什么不含 token

- baseSpec 是整个 JVM 共用的模板，一份就够
- token 是每个测试类自己的，不能进模板
- **分层设计**：不变的进模板，会变的走参数

#### unauth() 为什么要 given().spec(baseSpec)

不能直接 `return baseSpec`：**共享风险**：调用方 `.header(...)` 会改模板本身，影响所有人
- `given().spec(baseSpec)` 的含义："基于模板新开一个，你随便改"

#### 日志静默 + Allure 报告

```java
PrintStream silent = new PrintStream(PrintStream.nullOutputStream());
.addFilter(new RequestLoggingFilter(LogDetail.ALL, silent))
```

- 关掉控制台输出（CI 不刷屏）
- AllureRestAssured Filter 自动把请求/响应附加到报告
- 口诀：**关掉嘴（控制台），开着耳朵（报告）**

#### 依赖注入

```java
public RequestSpecFactory(Env env) { ... }
```

Env 通过构造函数传进来，好处：
- **可测试**：可以传假 Env 进来
- **可换环境**：-Denv 切换时传不同的 Env
- **没有全局状态**：不读静态字段就没有竞态

### 2.3 BaseApi — 业务 API 抽象基类

**关键知识点**：

#### 模板方法模式 vs 抽象基类

- **真·模板方法**：父类定流程（先A再B最后C），子类填细节
- **BaseApi**：父类给工具（`req(token)`），子类自由发挥
- 准确分类：**提供共享基础设施的抽象基类**，借鉴了模板模式"复用骨架代码"的思想

#### 职责分离

- API 类的职责：发请求 → 返回原始 Response
- 测试类的职责：拿到 Response → 做断言
- API 里不应写 `.then().statusCode(200)` 等断言逻辑

#### 为什么 req(token) 不直接暴露 factory

让子类 API 不需要知道 factory 的存在，只调用语义更明确的 `req(token)`。

### 2.4 BaseTest — 测试基类

**关键知识点**：

#### 为什么是 abstract

- 本身没有 @Test 方法
- 存在意义是给子类提供共享能力（token、specFactory、jdbcTemplate）
- abstract 明示"别直接用我，你应该继承我"

#### @TestInstance(PER_CLASS)

- JUnit5 默认每个 @Test 创建一个新测试类实例，@BeforeAll 必须是 static
- PER_CLASS 改为整个测试类只创建一个实例，@BeforeAll 可以是非静态方法
- **核心好处**：用实例字段代替静态字段，摆脱全局共享

#### protected 字段

| 修饰符 | 谁能访问 |
|---|---|
| public | 所有人 |
| protected | 同包 + 子类 |
| (默认) | 同包 |
| private | 只有本类 |

token/specFactory/jdbcTemplate 用 protected，让子类直接访问。

#### login() 设计为可覆盖

```java
protected String login() {
    return new AuthApi(specFactory).loginAndGetToken(env);
}
```

- 部分测试类需要不同账号登录（如测试权限差异）
- 子类可以 @Override 返回不同 token

#### DB 连接：DruidDataSource + 快速失效

```java
ds.setConnectTimeout(5000);   // TCP 握手超时 5 秒
ds.setSocketTimeout(10000);   // 单次 SQL 读写超时 10 秒
ds.setMaxWait(10000);         // 连接池拿连接最多等 10 秒
```

DB 不通时快速失败，而不是死等到 CI timeout 强杀。

#### 测试数据基础设施

```java
protected Long ensureTestOrg() { ... }
```

- 幂等插入测试机构，多个测试类可安全调用
- 测试机构是共享基础设施，不在 @AfterEach 中删除

### 2.5 JsonTemplateUtil — JSON 模板工具

**解决的问题**：把请求体存为 JSON 文件，测试时读取后按需改几个字段。

**关键知识点**：

#### 模板 + set/delete 的核心思路（v1 继承）

```java
JsonTemplateUtil.load("template/user/save.json")
    .set("$.username", "alice")     // 改字段值
    .delete("$.id")                 // 整个字段从 JSON 中消失
    .toJson();                      // 转 JSON 字符串
```

- `.set()` 改字段值（用于"值错误"场景）
- `.delete()` 彻底移除字段（用于"必填缺失"场景）
- delete vs set null：set null 保留字段但值为 null，delete 让字段完全消失，服务端校验结果不同

#### 流式 API（Fluent Interface）

每个操作方法返回 this，支持链式调用。关键是 `return this`。

#### v3 vs v1 的改进

| | v1 | v3 |
|---|---|---|
| 模板对象 | 父类共享字段 `defaultData` | 方法内局部变量，用完即弃 |
| 并发安全 | 方法级并发时 DocumentContext 互相污染 | 局部变量在线程栈上，天然隔离 |
| 封装 | 直接暴露 DocumentContext 底层 API | 封装成语义化的 set/delete/toJson/toMap |

**核心原则**：变化频率越高，作用域应该越小。

```
变化频率: 不变 < 每类一次 < 每方法一次 < 每次调用
作用域:   static 字段 < 实例字段 < 局部变量 < 方法参数
           对应关系: 左边匹配右边
```

#### toJson() vs toMap() — 强约束

| HTTP 方法 | 用法 | 说明 |
|---|---|---|
| POST/PUT | `.body(tpl.toJson())` | JSON 字符串，Content-Type: application/json |
| GET | `.queryParams(tpl.toMap())` | Map，序列化为查询参数 |

反例：
- POST 用 `toMap()` + `.body(map)` → 可能被当作 form 参数序列化
- GET 用 `toJson()` + `.body(json)` → GET 请求不该有 body

### 2.6 POJO 体系

#### 请求侧 vs 响应侧

- **请求侧**：全部用 JsonTemplateUtil + JSON 模板，不建请求 POJO
  - 理由：异常用例需要灵活删字段/改字段，模板比 POJO 灵活
- **响应侧**：按需建 POJO

#### 何时建响应 POJO

- ✅ 必须建：响应字段会被多个用例断言，或要从响应里取数据给后续用例
- ✅ 必须建：列表/分页接口（用 PageResult<T> 泛型壳）
- ❌ 不建：只断言 code/msg 的接口（删除、登出）
- ❌ 不建：一次性接口、字段超多但只断言 1-2 个的

#### POJO 规范

```java
@Data                                    // Lombok 生成 getter/setter/toString/equals/hashCode
@JsonIgnoreProperties(ignoreUnknown = true)  // 接口加字段后 POJO 不挂
public class CommonResp<T> {
    private Integer code;
    private String msg;
    private T data;
}
```

- 必须用 `@Data`
- 必须加 `@JsonIgnoreProperties(ignoreUnknown = true)`
- 字段名和后端 JSON 不一致时用 `@JsonProperty` 映射

## 三、泛型与类型擦除

### 3.1 泛型的作用

1. **编译期检查**：类型不对编译就报错，不等运行时崩
2. **不用强转**：取出来直接是正确类型
3. **代码可读**：`List<User>` 一眼知道装的什么

### 3.2 类型擦除

Java 的泛型是"伪泛型"——编译时严格，运行时擦除。

```java
List<String> stringList = new ArrayList<>();
List<Integer> intList = new ArrayList<>();
stringList.getClass() == intList.getClass(); // true
```

编译后 `List<String>` 和 `List<Integer>` 是同一个 ArrayList。

### 3.3 类型擦除导致反序列化问题

```java
CommonResp<SysTokenVO> resp = response.as(CommonResp.class);
resp.getData().getAccessToken();  // ClassCastException!
// data 实际是 LinkedHashMap，不是 SysTokenVO
```

Jackson 拿到 CommonResp.class 时，T 已被擦除，不知道 data 该反序列化成什么，默认转 LinkedHashMap。

### 3.4 TypeRef 解决方案 — 超类型令牌

```java
CommonResp<SysTokenVO> resp = response.as(new TypeRef<CommonResp<SysTokenVO>>() {});
//                                          注意后面有 {} ↑
```

`new TypeRef<...>() {}` 创建了 TypeRef 的**匿名子类**。

**精妙之处**：类型擦除有一个例外——**子类的父类泛型参数会被保留在字节码里**。

TypeRef 构造函数内部：
```java
Type superclass = getClass().getGenericSuperclass();  // 拿到 TypeRef<CommonResp<SysTokenVO>>
this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];  // 提取 CommonResp<SysTokenVO>
```

同类技术：Jackson 的 TypeReference、Spring 的 ParameterizedTypeReference，原理相同。

## 四、并发安全

### 4.1 JVM 内存模型基础

```
┌─────────────────────────────┐
│  方法区 / Metaspace          │  ← 类信息、static 字段
├─────────────────────────────┤
│  堆 (Heap)                  │  ← new 出来的对象，所有线程共享
├─────────────────────────────┤
│  栈 (Stack) - 每线程一个     │  ← 局部变量，天然线程安全
└─────────────────────────────┘
```

- **static 非 final 字段 + 多线程 = 大概率有 bug**
- **final 字段 = 安全**（JMM 保证构造函数结束后可见）
- **栈上的局部变量 = 天然安全**（每线程独享）

### 4.2 并发安全的三个思路

| 思路 | 安全性 | 说明 |
|---|---|---|
| 不共享 | ⭐⭐⭐⭐⭐ | 最简单最稳，每次 new 新的 |
| 不可变 | ⭐⭐⭐⭐ | 共享但只读，也安全 |
| 加锁同步 | ⭐⭐ | 能用但容易写错、性能差 |

**工程界共识**：能不共享就不共享。

### 4.3 v1 的并发隐患与 v3 的解决

| 隐患 | v1 写法 | v3 解决 |
|---|---|---|
| token 被覆盖 | `public static String token` | protected 实例字段 + PER_CLASS |
| RestAssured 全局配置 | `RestAssured.basePath = "..."` | RequestSpecFactory 实例化 spec |
| 模板对象共享 | 父类字段 `defaultData` | 方法内局部变量 |
| DB 全表清理 | `DELETE FROM sys_user` | 按前缀精确删除 |

### 4.4 JUnit5 并发粒度

```
类级别并发：不同类并行，同类内串行
  junit.jupiter.execution.parallel.mode.classes.default = concurrent
  junit.jupiter.execution.parallel.mode.default = same_thread

方法级别并发：同类内方法也并行
  junit.jupiter.execution.parallel.mode.default = concurrent
```

## 五、测试用例设计

### 5.1 覆盖标准 — 档位 2（基础回归）

每个接口至少：
- **1 个正向用例**：合法参数，验证业务正常
- **2-3 个异常用例**：凭证错误、必填缺失、未授权
- **可选边界用例**：超长/特殊字符，仅接口明确有需求时加

一个接口 5-8 个用例为宜，超过 10 个考虑过度设计。

### 5.2 不测什么

- ❌ 测试环境关闭的功能（如 Maku 的验证码）
- ❌ 纯前端校验
- ❌ 登录接口的"未授权访问"（它本来就不要 token）
- ❌ SQL 注入、XSS（除非明确安全测试需求）

判断标准：**服务端真的会基于这个字段做不同行为吗？不会就别测。**

### 5.3 测试代码四段式（强制）

```java
// ① 准备：用模板构造请求体
String body = JsonTemplateUtil.load("template/user/save.json")
        .set("$.orgId", testOrgId)
        .toJson();

// ② 调用：Api 类发请求，返回 Response
Response response = userApi.save(token, body);

// ③ 解析：用 POJO 解析响应（异常用例可省略）
CommonResp<SysTokenVO> resp = response.as(new TypeRef<CommonResp<SysTokenVO>>() {});

// ④ 断言：HTTP 层 + 业务层 + 字段层
response.then().statusCode(200);
assertThat(resp.getCode()).isEqualTo(0);
assertThat(resp.getData().getAccessToken()).isNotBlank();
```

异常用例省略③，直接 body 断言：
```java
response.then().statusCode(200)
        .body("code", not(equalTo(0)))
        .body("msg", equalTo("用户名不能为空"));
```

### 5.4 命名规范

```
should_<期望>_when_<条件>
```

| 场景 | 命名 |
|---|---|
| 正向 | `should_return_token_when_valid_credentials` |
| 凭证错误 | `should_fail_when_invalid_credentials` |
| 必填缺失 | `should_fail_when_missing_required_field` |
| 未授权 | `should_return_401_when_no_token` |

## 六、断言粒度规则

### 6.1 正向响应（code=0）

- ✅ 必测 HTTP 200
- ✅ 必测 code = 0
- ❌ **不测 msg 文案**（通常是 "success"，没业务含义，测了是脆弱测试）
- ✅ 测 data：
  - data=null：跳过，改用 DB 验证
  - data 有结构：JSON Schema 校验结构 + 关键字段值

### 6.2 异常响应（code≠0）

- ✅ 必测 HTTP 200（Maku 业务错误也返回 200）
- ✅ 必测 code 非 0
- ✅ **必测 msg 精确文案**（用户能感知的提示信息，需要保护）
- ❌ 不测 data（通常是 null）

### 6.3 DB 验证（增删改必做）

| 操作 | DB 验证 |
|---|---|
| save | 应能查到新增记录，关键字段对得上 |
| update | 目标字段应已变化 |
| delete（软删除） | deleted=0 的记录应不存在 |
| 查询/登录 | 跳过 DB 验证 |

## 七、数据驱动

### 7.1 CSV 文件组织

**一个 CSV = 一种参数处理逻辑**：

| 文件名约定 | 处理方式 | 测试方法 |
|---|---|---|
| `<动作>_invalid_<字段>.csv` | `set` 改字段值 | should_fail_when_invalid_xxx |
| `<动作>_missing_field.csv` | `delete` 删字段 | should_fail_when_missing_required_field |
| `<动作>_boundary.csv` | `set` 边界值 | should_fail_when_boundary_xxx |

**反例**：改字段和删字段混在一个 CSV → 测试方法要写 if-else，代码丑且易错。

### 7.2 CSV 列设计

- 第一列必须是 `description`（人类可读的场景描述）
- 后续列是用例参数 + 期望结果
- null（空单元格）→ `.delete()` 删字段；非 null → `.set()` 改字段

### 7.3 模板和 CSV 的分工

- **模板管"全集"**：按接口文档列全字段，默认值跑通正向
- **CSV 管"差异"**：只列本次要变化的字段，不变的字段由模板兜底

### 7.4 请求模板完整性要求

- 必须包含接口文档中定义的所有请求参数（包括可选字段）
- 字段默认值与接口文档的请求示例保持一致
- 理由①：模板是接口文档的可执行版本，缺字段等于遮蔽了接口能力
- 理由②：异常用例依赖 `.delete()` 构造"字段缺失"场景，模板里不存在的字段删不了

## 八、CI/CD

### 8.1 核心概念

- **CI（持续集成）**：代码每次提交，自动跑测试，验证改动有没有破坏功能
- **CD（持续部署/交付）**：测试通过后，自动部署到环境

### 8.2 三种触发场景

| 触发源 | 场景 | 价值 |
|---|---|---|
| 被测系统代码变更 | 后端 push → 部署 → 触发回归 | 发版门禁 |
| 测试代码变更 | 测试 push → 跑测试 + 部署报告 | 代码保护 |
| 定时触发 | 每天凌晨 2 点全量跑 | 持续监控 |

### 8.3 流水线阶段

```
拉代码 → 静态检查 → 编译构建 → 单元测试 → 构建制品
→ 推制品到仓库 → 部署测试环境 → 自动化测试 → 部署 staging
→ 人工审批 → 部署生产 → 健康检查/回滚
```

### 8.4 流水线设计原则

- **Fail Fast**：快速失败，提早暴露问题
- **可重复**：任何人在任何机器上跑结果一致
- **可观测**：每一步都有日志和指标
- **幂等性**：重复跑同样的输入，结果一样
- **分支隔离**：特性分支不影响主干

## 九、接口文档与实战经验

### 9.1 接口文档现状

| 公司规模 | 文档质量 |
|---|---|
| 大厂 | 工具自动生成 + 开发手动补全，有专门 API 平台 |
| 中型 | 工具生成但开发者不爱写注释，文档残缺 |
| 小公司 | 靠测试自己摸，口头沟通 |

**结论**：80% 的接口文档不完整，这是常态。

### 9.2 文档不全时的补充方法

1. 抓包手动整理：正向跑一遍记录响应结构，异常一个个试记录响应
2. 翻后端代码核对（有权限时）
3. 找最熟悉模块的开发问 5 分钟

### 9.3 看不到后端代码的经验

- Spring Boot 项目几乎都用同一套校验（@NotNull/@NotBlank/@Size）
- CRUD 接口的"套路"是固定的（新增 7 个关注点、修改 6 个、删除 5 个）
- 同类型字段的反应高度相似（String 必填测一个代表全部）
- 学会一个接口的错误模式，其他接口大概率一样

## 十、v1 / v2 知识点（历史经验）

### 10.1 v1 的核心经验（保留）

- ✅ JsonPath 模板 + `.set()` / `.delete()` 改字段做异常用例
- ✅ 按模块组织 template/params/schemas 三个目录
- ✅ JdbcTemplate 做数据库结果验证和清理
- ✅ 异常场景覆盖意识（必填项缺失、未授权、边界值）
- ✅ Schema 结构断言（matchesJsonSchemaInClasspath）
- ✅ Allure 注解 @Feature/@Story/@DisplayName

### 10.2 v1 的并发问题（已修复）

| 问题 | 原因 | 影响 |
|---|---|---|
| `static String token` | 方法区只有一份，所有线程共享 | 并发时互相覆盖，401 错误 |
| `RestAssured.basePath = "..."` | 修改全局静态变量 | 并发时请求发到错误路径 |
| `DELETE FROM sys_user` | 全表清理 | 一个进程清了数据，另一个进程验证失败 |

### 10.3 v1 模板参数处理的问题

```java
// v1: 模板对象做父类共享字段
protected DocumentContext defaultData;

// 即使单线程也有隐患：
// - DocumentContext 是可变对象，.set()/.delete() 改的是它本身
// - @BeforeEach 重置解决了单线程问题，但方法级并发仍有竞态
```

### 10.4 v1 CSV 组织的问题

- 改字段和删字段混在一个 CSV
- 测试方法里要写 if-else 判断"这条是改字段还是删字段"
- v3 拆开后代码量减半

### 10.5 v2 定位

v2 是简化版，作为 v3 的起点。v1 是业务逻辑的"参考资料"，特别是鉴权、加签、特殊 Header。任何来自 v1 的业务规则必须显式确认后迁移，不能"看起来对就抄"。
