# 分支管理实践

> 本文结合 **ReqMakuTest 毕设项目**的实际情况，介绍主流工作实践。
> 当前仓库状态：本地 `main`，远端 `origin/main` + `origin/master`（历史遗留）。

---

## 1. 核心原则

| 原则 | 说明 |
|------|------|
| **main 永远可用** | main 上的每一个 commit 都应该是可编译、测试通过的稳定版本 |
| **工作在分支上进行** | 不直接在 main 上开发，改动完成并验证后再合并回来 |
| **小步提交，频繁合并** | 一个阶段做完就合并，不要让分支离 main 太远 |

---

## 2. 本项目推荐的分支结构

```
main          ← 稳定主干，每个阶段完成后合并一次
  │
  └── dev     ← 日常工作分支，在这里写代码、提交、调试
```

### 为什么是这两个分支，而不是更多？

这是一个**个人学习型项目**，不需要 GitFlow 那样的 `develop / release / hotfix` 多分支体系。
两个分支已经能做到：
- main 始终保存"上一个里程碑完成时"的干净状态（方便导师/答辩时查看）
- dev 是你每天工作的地方，可以大胆实验，不用担心污染主干

> 如果某个阶段的实验性改动比较大（比如你想同时尝试两种架构方案），
> 可以再从 dev 拉一个临时分支 `feat/xxx`，实验完选好再合并到 dev。

---

## 3. 处理当前的 master/main 问题

远端有两个主干（`origin/master` 和 `origin/main`）是历史遗留，保留其中一个即可。
**保留 `main`**（GitHub 现代默认），忽略 `origin/master`，不必主动删它。

## 4. 日常工作流（学习阶段怎么用分支）

### 4.1 开始工作前

```bash
# 确保 main 是最新的
git checkout main
git pull origin main

# 切到工作分支（第一次需要创建）
git checkout -b dev        # 第一次：创建并切换
git checkout dev           # 之后：直接切换
```

### 4.2 开发中（在 dev 分支上）

```bash
# 正常写代码，随时提交
git add -A
git commit -m "feat: xxx"

# 推送 dev 到远端（备份 + 老师可以看到过程）
git push origin dev
```

### 4.3 一个阶段完成后，合并到 main

```bash
# 确认 dev 上测试通过（mvn test 全绿）后
git checkout main
git merge dev --no-ff -m "merge: 阶段X完成 - [一句话概括]"
git push origin main
```

> `--no-ff`（no fast-forward）的作用：
> 保留一个合并节点，让 git log 能清晰看到"哪些 commit 属于哪个阶段"，
> 比全部线性历史更容易回溯。

### 4.4 合并后继续下一阶段

```bash
# 不需要删 dev，直接在 dev 上继续
git checkout dev
# 开始写阶段2的代码...
```

## 5. 对应本项目各阶段的操作节点

| 节点      | 操作 | 备注 |
|---------|------|------|
| 阶段1完成 ✅ | 已在 main 提交（d73c344） | 已完成 |
| 开始阶段2   | `git checkout -b dev` | 从 main 拉出 dev 分支 |
| 阶段2完成   | 在 dev 跑通后 merge 到 main | Auth 模块 |
| 阶段3完成   | 同上 | User 模块 |
| 阶段4完成   | 同上 | 其他模块批量迁移 |
| 长期阶段    | 确保 main 是最新可运行状态 | main 是展示用的 |

## 6. Commit Message 规范（conventional commits）

本项目按 CLAUDE.md 要求使用 conventional commits：

```
<类型>(<范围>): <简短描述>

<可选详细说明>
```

| 类型 | 使用场景 | 示例 |
|------|---------|------|
| `feat` | 新功能/新用例 | `feat: 新增 AuthLoginTest 正向用例` |
| `fix` | 修 bug | `fix: 修复 token 并发安全问题` |
| `refactor` | 重构（不改功能） | `refactor: 提取分页断言为公共方法` |
| `test` | 补充测试 | `test: 补充 UserSave 边界值用例` |
| `chore` | 配置/构建/依赖 | `chore: 升级 RestAssured 到 5.4.0` |
| `docs` | 文档 | `docs: 更新分支管理说明` |

## 7. 快速参考：常用命令

```bash
# 查看当前在哪个分支
git branch

# 查看所有分支（含远端）
git branch -a

# 查看提交历史（图形化）
git log --oneline --graph --all

# 切换分支
git checkout <分支名>

# 创建并切换
git checkout -b <分支名>

# 推送到远端（首次）
git push -u origin <分支名>

# 推送到远端（后续）
git push origin <分支名>
```

## 8. 可视化分支历史长什么样

完成几个阶段后，`git log --oneline --graph --all` 应该看起来像这样：

```
*   a1b2c3d (HEAD -> main, origin/main) merge: 阶段3完成 - User模块
|\
| * 9f8e7d6 (dev) test: UserDeleteTest 全部场景
| * 5c4b3a2 feat: 新增 UserSaveTest
| * 1d2e3f4 feat: 新增 UserPageTest
|/
*   7g8h9i0 merge: 阶段2完成 - Auth模块
|\
| * ...
|/
* d73c344 feat: 阶段1工程骨架搭建完成    ← 你现在在这里
```

这样的历史对毕设答辩很友好：每个阶段的工作一目了然。
